# View Layer

## 目的

ViewをMonoBehaviourベースの表示層として扱い、受け身な表示責務を軸にしながら、Unity都合で必要な場合だけ最小限のドメイン責務を扱える形で実装する。

## Viewの本質

ViewはMVRPのViewに相当する。基本的にドメインはModelに閉じ込め、Viewは表示、入力通知、表示アニメーション、Unityオブジェクトとの接続を担当する。ただし、Unity依存が強くModelへ閉じ込めづらいドメイン知識や、Model層に置くメリットが薄い処理はViewへ置くことを許容する。理由があるときだけ責務を広げ、通常は受け身なView実装を優先する。

## ルール

- `MonoBehaviour` は基本的に継承する。
- 他の層への依存は `Model` に対してのみ許可する。
- 公開I/Fは通知イベント中心に設計し、必要時のみドメイン寄りメソッドを追加する。
- ドメイン判断は、特別な理由がなければDirector/Modelに委譲する。
- 表示の知識はViewに閉じ込める。Directorに表示やアニメーションの詳細を持ち込まない。
- 複数のクラスが利用するView生成責務はFactoryクラスとして分離する。ひとつのクラスが利用するだけならひとまずそのクラスに置く。
- Viewとして振る舞うクラスは、基本的に `ViewBase` を継承する。表示/非表示アニメーションを統一的に管理するための基底クラスである。ただし、ヘルパークラスなど通常のView責務とは異なるクラスはその限りではない。
- アニメーション実装が必要なら `$litmotion` を併用する。
- Inspector公開は調整が必要な値に限定する。
- イベントは `Observable` などで外へ通知する。
